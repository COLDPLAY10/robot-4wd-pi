# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Что это

Автономный робот на платформе Yahboom Raspbot V2 (4WD-мекалум, Raspberry Pi 5): SLAM по лидару Yahboom T-MINI Plus, depth-восприятие камерой (Depth-Anything-V2 ONNX), ультразвук, A*-планирование, реактивный объезд препятствий. Код и комментарии — на русском, придерживайся этого.

Тестов и линтера в проекте нет. Запуск целиком возможен только на Pi: `car_adapter.py` импортирует `Raspbot_Lib` (ставится отдельно по инструкции Yahboom) на уровне модуля, поэтому на дев-машине всё, что тянет `car_adapter`/`navigation_controller`, не импортируется. Без железа можно импортировать и проверять: `map/slam_core.py`, `map/scan_matcher.py`, `map/path_planner.py`, `map/sensor_fusion.py`, `map/map_visualizer.py`, `camera_perception/*`. У `scan_matcher` и `path_planner` есть numpy-fallback'и на случай отсутствия cv2.

## Команды

```bash
pip3 install -r requirements.txt          # numpy, pyserial, opencv, onnxruntime, matplotlib

# Основной сценарий (на роботе, из map_scripts/):
python3 demo_with_lidar.py explore                      # построить карту → .pkl + .png по Ctrl+C
python3 demo_with_lidar.py explore --map old.pkl        # дополнить существующую карту
python3 demo_with_lidar.py goto X Y --map map.pkl       # ехать к цели по готовой карте (localization)

python3 autonomous_navigation.py                        # интерактивный режим (explore/goto/save/load)
python3 demo_no_lidar.py explore                        # деградированный режим: УЗ + камера

# Утилиты:
python3 diagnose_lidar.py                               # диагностика UART/лидара
python3 map_scripts/calibrate_odometry.py               # калибровка MAX_LINEAR/ANGULAR_SPEED
python3 map_scripts/visualize_map.py map.pkl            # перерисовать карту
python3 map_scripts/compare_trajectory.py map.pkl       # дрейф одометрии vs SLAM, loop-closure
python3 scripts/setup_depth_model.py                    # экспорт Depth-Anything-V2 → ONNX (на дев-машине, нужны torch+transformers)
python3 scripts/setup_segformer_model.py                # экспорт SegFormer-B0 → ONNX (на дев-машине; печатает id2label и индекс класса floor)
python3 scripts/eval_floor_segmentation.py              # офлайн-оценка сегментации пола + автокалибровка камеры по кадрам заезда
```

Модели лежат в `models/` (не в git): `depth_anything_v2_small.onnx` (восприятие глубины) и `segformer_b0_ade.onnx` (сегментация пола).

## Архитектура

Поток данных:

```
lidar.py / ultrasonic.py / камера+DepthEstimator
        │ (NavigationController._update_sensors, ~20 Гц)
        ├──► SLAM (map/slam_core.py) ──► OccupancyGrid ──► PathPlanner (A* по inflated grid)
        └──► SensorFusion (реактивный слой, секторные дистанции front/left/right/back)
                    │
        NavigationController (конечный автомат) ──► car_adapter.py ──► PWM моторов
```

Два контура работают параллельно и не надо их путать:
- **Картографический** (SLAM → A*): медленный, глобальный, через occupancy grid.
- **Реактивный** (SensorFusion): мгновенные секторные расстояния для стопов/объезда, в карту не пишет.

### SLAM (`map/slam_core.py`)

- `OccupancyGrid` 400×400 × 0.05 м (20×20 м), значения 0..100, prior 50, обновление log-odds с клампом ±5. Старт робота = центр карты = мировое (0,0).
- **Одометрия открытого контура**: энкодеров нет, скорость оценивается из PWM (`MAX_LINEAR_SPEED`, `MAX_ANGULAR_SPEED` в `NavigationController`). Ошибка ±25% и хуже — точность позы целиком держится на scan matcher'е.
- **Scan matcher** (`map/scan_matcher.py`, Hector-style): likelihood field = distance transform от occupied-ячеек (порог 65), Gauss-Newton (≤14 итераций, ранний выход по сходимости) по билинейно интерполированному полю. Гейты отказа: residual > 0.18 м, прыжок позы > 0.8 м / 45° от одометрического прайра, мало точек в «виденной» области. При отказе матча поза остаётся одометрической — это штатно.
- Параллельно ведётся `odom_only_history` (чистая одометрия без коррекций) — для отчётных сравнений, не трогать при правках логики.
- **Режимы**: `mapping` (пишем карту + правим позу) и `localization` (карта из .pkl read-only, робот физически ставится в (0,0,0), `reset_pose=True`). Все `update_with_*` проверяют режим перед записью в карту.
- Запись в карту: лидар (occupied 0.9 / free вдоль луча 0.5), ультразвук (конус ±15° семью лучами), камера (depth → обратная pinhole-проекция `camera_perception/projection.py` → фильтр по высоте 3–50 см над полом → occupied 0.7). `update_with_camera_depth` ВСЕГДА возвращает облако препятствий, даже в localization — оно нужно реактивному слою.

### Приоритеты сенсоров (`map/sensor_fusion.py`)

Иерархия доверия: **лидар (0.9) > камера (0.7) > ультразвук (0.6)**, таймаут данных 2.5 с (у камеры свой — 1.5 с). Логика `get_obstacle_distance`:
- База = лидар (третья ближайшая точка сектора — робастный минимум; точки <12 см отбрасываются как шум) ⊕ ультразвук (только front). Конфликт «лидар <0.2 м, УЗ >0.4 м»: близкие точки в конусе УЗ ±15° (`front_narrow`) → шум лидара, верим УЗ; вне конуса → реальный объект сбоку, верим лидару. Иначе — **`min(лидар, УЗ)`**: УЗ, как и камера, видит низкие/тонкие препятствия (нога, порог), которые 2D-лидар пропускает, поэтому может только СОКРАТИТЬ фронт, а не «усреднить вверх» (прежнее 0.6·лидар+0.4·УЗ топило близкое показание УЗ — 2.0/0.15→1.26 — и робот въезжал).
- **Принцип безопасности камеры**: по фронту берётся `min(база, камера)` — камера может только *сократить* дистанцию (она видит низкие препятствия, недоступные 2D-лидару), но никогда не объявляет «свободно» в обход лидара. По бокам камера — только gap-fill при отсутствии лидара. Отключение реактивной камеры одним флагом: `CAMERA_REACTIVE_ENABLED` в `NavigationController`.
- **Коридорная метрика камеры** (`obstacle_distances_by_sector`, `corridor_halfwidth_m`): кроме угловых секторов, фронт-дистанция учитывает точки в полосе шириной робота независимо от пеленга — ловит угловые препятствия «под колесо» и диагональные стены. Дальности камеры раздельные: анализ сцены до `CAMERA_MAX_RANGE_M` (6 м), запись в карту до `CAMERA_MAP_MAX_RANGE_M` (3 м) — дальняя монокулярная глубина шумит, а occupied-ячейка блокирует A*.
- **Страховочный канал `nearest_in_depth_band`**: «ближайший объект по курсу» прямо из depth-карты, без проекции и фильтра высот — робастный минимум в полосе у линии горизонта (±5% кадра, центральные 55% ширины; пол в этих строках физически дальше ~3 м, ложных стопов от пола нет). Ловит сетки/прозрачное/редкие облака, где секторный канал молчит. Вливается в `front` камеры через min.
- **Отладка камеры**: `CAMERA_DEBUG_SAVE_ENABLED` в `NavigationController` — раз в `CAMERA_DEBUG_PERIOD_S` пишет в `results/<запуск>/depth/` кадр+depth с красной маской пикселей-препятствий (`camera_perception/debug_viz.py`); ротация `CAMERA_DEBUG_KEEP`.
- **Выбор направления**: при препятствии/объезде контроллер поворачивает в сторону большего простора (`_choose_turn_direction`/`_side_clearance` по секторам SensorFusion), а не всегда налево.

### Inflation (`map/path_planner.py`)

- Перед планированием препятствия (ячейки ≥ 65) раздуваются `cv2.dilate` эллипсом радиуса `ROBOT_RADIUS + SAFETY_MARGIN` (0.12 + 0.08 = 20 см), throttle пересчёта 0.5 с.
- A* (8-связный) и проверка коллизий DWA ходят по inflated grid **центром робота** — размер робота уже учтён в раздувании, никаких проверок по окружности.
- Стартовая клетка в inflated-зоне разрешена (`allow_inflated=True`) — робот там физически стоит, off-by-one пиксель не должен ронять планирование. Целевая — нет.
- Тот же радиус рисуется красным поясом на PNG карты (`save_map` → `map_visualizer.render_map_png`) — визуализация обязана совпадать с тем, по чему реально планирует A*.

### Навигация (`map/navigation_controller.py`)

Конечный автомат: `IDLE → EXPLORATION | NAVIGATION → OBSTACLE_AVOIDANCE`. Каждый `update()` → `_tick()` (сенсоры + одометрия + SLAM, dt capped 0.5 с) → поведение режима:
- **EXPLORATION** — реактивное блуждание: время езды вперёд пропорционально дистанции до препятствия; <0.4 м — поворот; <`critical_distance`=0.28 м — экстренный стоп + назад (порог прерывания ПОВОРОТА отдельный и теснее — `rotate_safety_distance`=0.20, чтобы робот не робел отворачиваться от близкого объекта); `consecutive_rotations` (макс 3) разрывает зацикливание. Реактивная проверка в `_safe_move_forward` идёт ДО `sleep` (нет слепого хода в начале сегмента), интервал 0.1 с.
- **NAVIGATION** — **DWA** (`_dwa_navigation_step`, неблокирующий такт): робот непрерывно течёт по дугам к текущему waypoint'у A*. Команда (v, ω) → танковые PWM бортов (`ca.move_tank`, команда одометрии `tank`) с квантованием мёртвой зоны моторов (`MIN_MOVE_PWM`). Реактивный слой — страховка поверх: фронт < `DWA_BLOCK_FRONT_M` → стоп + OBSTACLE_AVOIDANCE (критично в localization, где свежие препятствия в карту не пишутся); фронт ограничивает скорость DWA через `v_max_cap`. Застой (нет траектории или команда заквантовалась в 0) `DWA_STALL_LIMIT` тактов → перепланирование A*. Откат на старый дискретный автомат одним флагом `DWA_ENABLED=False` (`_legacy_navigation_behavior`).
- **OBSTACLE_AVOIDANCE** — назад → объезд через свободную сторону из SensorFusion → перепланирование A*; неудача → EXPLORATION.

DWA в `path_planner.compute_velocity_command` векторизован (все кандидаты окна симулируются разом, ~1 мс на Pi): коллизии — по НЕПРЕРЫВНОМУ полю расстояний (`_clearance_field`), не по бинарному inflated grid — у границ раздувания бинарная проверка дедлочит (escape-правило: из тесного места можно выскальзывать, нельзя углубляться). Упрощение A*-пути — line-of-sight (`_segment_free`); прежний вариант (отклонение от хорды к соседу) схлопывал ЛЮБОЙ путь в прямую сквозь стены.

Все движения — через `_safe_move_forward`/`_safe_rotate`, которые обязаны тикать `_tick()` каждые 0.1–0.2 с, иначе одометрия не накапливается. При добавлении любого блокирующего движения сохраняй этот паттерн: `ca.команда()` → `_set_movement_command()` → `self.last_update_time = time.time()` → цикл с `_tick()`.

Плавность движения (важно не сломать):
- **Инференс глубины — в фоновом потоке** (`_DepthWorker`): главный цикл НЕ блокируется на нейронке, `_update_sensors` неблокирующе забирает последний результат; поза для проекции облака снимается в момент захвата кадра. Не возвращай инференс в `_tick()`.
- **Круиз**: в EXPLORATION `_safe_move_forward(..., stop_at_end=False)` не останавливает моторы между последовательными «вперёд»; стоп — только при препятствии/повороте. `_safe_rotate` и blocked-ветка сами останавливают робота, если он на ходу.
- **`forward_steer`**: подруливание дифференциалом бортов (`ca.move_param_forward`, p>0 → правый борт → влево; при ЛЮБОМ знаке p один борт УСИЛИВАЕТСЯ). Одометрия моделирует его в `_commanded_velocity` (v×(1+|p|/200), ω=v·p/100/WHEEL_BASE) — любое новое «кривое» движение обязано получить такую же модель, иначе scan matcher теряет прайр. Используется: увод к простору на круизе (`_steer_to_free_space`, только если обе стороны измерены) и докрутка курса в legacy-NAVIGATION.
- **Запись в карту гейтится**: лидар пишется только при НОВОМ обороте (`revolution_count`) и |ω| ≤ 0.6 рад/с (точки оборота копятся ~150 мс и при вращении мажутся), УЗ — не чаще 2 Гц; векторная запись `update_cells_batch` — лидар И камера-глубина (`update_with_camera_depth`); поштучный `update_cell` ел 0.1–0.4 с на тик (УЗ-конус из 7 лучей мал, остаётся поштучным). Известное ограничение: free-лучи лидара постепенно стирают камерные «низкие» препятствия из карты — защита от них в реактивном слое, не в A*.
- **Запись кадров — в фоне** (`_ImageLogger`): JPEG+depth(`.npy`) в `results/<запуск>/segmentation/` пишет отдельный поток-очередь; лог отвязан от инференса (кадр пишется, даже если глубина недоступна), каталог проверяется на запись на старте. Раньше синхронный `cv2.imwrite`+`np.save` прямо в `_update_sensors` морозил контур ~50–150 мс/кадр — НЕ возвращай запись на горячий путь. Флаг `FLOOR_SEG_CAPTURE_RAW`.
- **Слияние, краевые случаи**: кластер ≥5 под-12см точек лидара в секторе = прижатый объект (0.10 м), а не «пусто»; конфликт «лидар близко/УЗ далеко» решается в пользу УЗ только если близкие точки в его конусе ±15°; секторная дистанция = третья ближайшая точка (робастный минимум, медиана квартиля растворяла ножки стульев); у камеры свой таймаут 1.5 с + сброс после поворота (`invalidate_camera`). Отъезды назад — через `_safe_backward` (контроль сектора 'back').
- **Плавное замедление**: `_speed_for_distance` — линейный спад PWM от 0.7 м до `MIN_MOVE_PWM=22` у 0.4 м.

### Константы — single source of truth

`WHEEL_BASE`, `ROBOT_RADIUS`, `SAFETY_MARGIN`, `MAX_LINEAR/ANGULAR_SPEED`, параметры камеры — определены в `NavigationController` и явно прокидываются в SLAM и PathPlanner при инициализации. `WHEEL_BASE` критичен: SLAM восстанавливает omega из скоростей бортов тем же значением, рассинхрон ломает сходимость scan matcher'а. Меняй только в одном месте.

## Что важно знать о состоянии кода

- **Лидар — YDLidar T-mini Plus** (пакеты `AA 55`, команды `A5 60`/`A5 65`), а НЕ Oradar MS200, как считал PR #7. `lidar.py` — рабочий драйвер с честным парсером протокола. Калибровочные константы (`DISTANCE_DIVISOR`, `CLOCKWISE`, `ANGLE_OFFSET_DEG`) проверяются на железе тестом `python3 lidar.py`. **`DISTANCE_DIVISOR=4` подтверждён** (тест у стены 1 м + дампы): на этом экземпляре сырое значение идёт в 1/4 мм для ОБОИХ форматов сэмпла, поэтому авто-дефолт для 3-байтного формата = 4.0 (раньше был 1.0 → дистанции вчетверо завышены, робот видел стены вчетверо дальше и бился об них). Протокол и разбор карт-«колец» от старого парсера — `samples/ANALYSIS.md`.
- **OpenCV на роботе — хрупкая смешанная установка** (`~/.local/.../site-packages`): бинарь `cv2.abi3.so` от `opencv-contrib-python` (его RPATH → `opencv_contrib_python.libs`), а `.libs`/dist-info от `opencv-python` (`opencv_python.libs`). Если `import cv2` падает `ImportError: libpng16-…so: cannot open shared object file` (камера мертва, кадры не пишутся, ошибка каждый тик в сенсоре 'camera') — восстановить симлинком `opencv_contrib_python.libs → opencv_python.libs` в site-packages. Это файловый фикс на SD робота, в git его нет.
- **DWA подключён и работает** в режиме NAVIGATION (см. раздел «Навигация»); прогнан на замкнутых симуляциях (перегородка/столб/Г-стена/разворот). Откат — `DWA_ENABLED=False`.
- **`camera_segmentation/` — переписан на нейросетевую сегментацию пола SegFormer-B0 (ADE20K), ONNX, v2.0.0** (старый цветовой `CameraSegmentation`/`simple_color_segmentation` удалён — в навигации не использовался). Выдаёт бинарную floor-маску (`FloorSegmenter`) + автокалибровку наклона/высоты камеры подгонкой плоскости по floor-пикселям (`floor_calibration.estimate_floor_plane`). **В боевой контур НЕ интегрирован** (в карту и реактивный слой не пишет) — это отдельный будущий этап, сейчас тестируется. Два флага в `NavigationController`: `FLOOR_SEG_CAPTURE_RAW=True` (дёшево писать сырые кадры RGB+глубина в `results/<запуск>/segmentation/` для офлайн-оценки, модель на роботе не нужна) и `FLOOR_SEG_ENABLED=False` (гонять SegFormer+калибровку прямо на роботе — тяжело, вторая нейросеть уполовинит FPS depth). Основной сценарий — офлайн: снять кадры заездом → `scripts/eval_floor_segmentation.py` на дев-машине (CSV по кадрам + paste-ready `CAMERA_MOUNT_TILT_RAD`/`CAMERA_MOUNT_HEIGHT_M`). Экспорт модели — `scripts/setup_segformer_model.py` (индекс класса `floor` зависит от чекпойнта, прописывается в `config.FLOOR_CLASS_IDS`). `update_with_camera_segmentation` в `slam_core` — устаревший хук, оставлен для совместимости со старыми скриптами.
- **Все артефакты запуска — в `results/<запуск>/`** (единый каталог, считается в `NavigationController.__init__`): `maps/` (карты `.pkl`/`.png`), `depth/` (отладка depth-восприятия), `segmentation/` (кадры для офлайн-оценки сегментации), `sensors/` (сырой лог сенсоров). `map/sensor_recorder.py` (`SensorRecorder`, включён по умолчанию) неблокирующе пишет `lidar_raw.bin` (сырые байты порта лидара для реплея через парсер драйвера — такой тест поймал бы баг «колец» сразу) и `events.jsonl` (сканы/УЗ/команды/позы/отметки кадров камеры на единой шкале времени) для офлайн-реплея и будущих тестов; при переполнении очереди дропает события, чтобы НИКОГДА не стопорить контур управления и поток лидара.
- Корневые `main.py`, `obstacle_detector.py`, `drive_ultrasonic.py` — старые реактивные скрипты без SLAM.
- README в `map_scripts/` местами устарел (упоминает сегментацию и несуществующие флаги `--lidar`; в argparse — `--no-lidar` и т.п.).
- Карты сохраняются как `.pkl` (грид + траектории + статистика) + `.png`; формат `.pkl` поддерживает старые файлы без `odom_only_history`.
