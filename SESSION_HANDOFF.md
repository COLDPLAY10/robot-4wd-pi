## 1. Что за проект

Автономный робот: платформа **Yahboom Raspbot V2** (4WD мекалум, ~1 кг),
**Raspberry Pi 5**, лидар **YDLidar T-mini Plus** (2D), USB-камера +
**Depth-Anything-V2** (моно-глубина, ONNX), ультразвук. Стек самописный на Python.
Цель — дипломная работа: SLAM-картирование + автономная навигация к точке с
объездом препятствий. Код и комментарии на русском (придерживаться).

Два параллельных контура (не путать): **картографический** (SLAM → occupancy
grid → A*) и **реактивный** (sensor_fusion → мгновенные секторные дистанции для
стопов/объезда, в карту не пишет).

Запуск только на Pi (`car_adapter` импортирует `Raspbot_Lib` на уровне модуля).
На дев-машине импортируются и тестируются: `map/slam_core`, `map/scan_matcher`,
`map/path_planner`, `map/sensor_fusion`, `map/map_visualizer`, `camera_perception/*`.
Для распаковки .pkl карт нужна заглушка: `sys.modules['Raspbot_Lib']=types.ModuleType(...)`.

Тестов/линтера в проекте нет. Всю проверку этой сессии делали офлайн на
синтетических данных + многоагентные ревью.

## 4. Лидар — драйвер YDLidar T-mini Plus (`lidar.py`)

Протокол (пакеты переменной длины):
`AA 55 | CT | LSN | FSA(2) | LSA(2) | CS(2) | LSN×сэмплы`.
Углы точек интерполируются между FSA и LSA (`(raw>>1)/64` градусов), дистанция из
сэмплов, контрольная сумма XOR, сборка полного оборота по `CT&1`, авто-детект
размера сэмпла (2 или 3 байта). Команды старта `A5 60`/`A5 52`/`A5 65` оставлены
из прежнего драйвера (с ними лидар реально запускался). API `LidarDriver`
сохранён 1:1. `get_scan()` отдаёт последний ПОЛНЫЙ оборот (до первого — пусто,
чтобы частичный сектор не читался как «простор»), `revolution_count` растёт на
каждый оборот.

**Калибровочные константы (НЕ проверены на железе)** — в классе `LidarDriver`:
- `DISTANCE_DIVISOR = None` (авто: 2-байтный сэмпл → 4.0, 3-байтный → 1.0).
  Проверка: робот в 1 м от стены, `python3 lidar.py --port /dev/ttyUSBx` — тест
  печатает медиану фронта при разных делителях, выбрать совпавший с рулеткой.
- `CLOCKWISE = True` — если карта зеркальная, поставить False.
- `ANGLE_OFFSET_DEG = 0.0` — если карта повёрнута, угол установки лидара.
- `MIN_RANGE_M=0.02`, `MAX_RANGE_M=16.0`.

Порт/скорость подобрать `python3 diagnose_lidar.py` (ожидаемо `/dev/ttyUSB1 @ 230400`;
`ttyUSB0` — голосовой модуль CH340, не лидар).

## 5. Физические параметры (замеры пользователя) — `NavigationController`

```
ROBOT_WIDTH      = 0.16583   # полная ширина по колёсам, м
TRACK_WIDTH      = 0.140     # колея (центры левого/правого колеса), м — ЗАМЕР
WHEEL_BASE_LONG  = 0.1172    # межосевое (центры передних/задних), м
ROBOT_LENGTH     = 0.20874   # нос камеры → задний край колеса, м
ROBOT_MASS_KG    = 1.0       # справочно
MOTOR_TORQUE_NM  = 0.8       # справочно
MOTOR_RPM_NO_LOAD= 245.0     # об/мин без нагрузки, ±10%
WHEEL_DIAMETER   = 0.060     # ⚠️ ТИПОВОЕ, НУЖНО ЗАМЕРИТЬ штангенциркулем
WHEEL_BASE = TRACK_WIDTH + WHEEL_BASE_LONG = 0.2572  # эфф. база диф-модели мекалум
MAX_LINEAR_SPEED = 0.6       # м/с при PWM=255 (уточнить калибровкой)
MAX_ANGULAR_SPEED= 4.7       # рад/с при PWM=255 (уточнить калибровкой)
ROBOT_RADIUS=0.13, SAFETY_MARGIN=0.08  # inflation = 21 см
```

**WHEEL_BASE для мекалума = колея + межосевое** (кинематика поворота на месте
ω=2v/(track+base)), а НЕ расстояние между бортами. Это же значение прокидывается
в SLAM — рассинхрон ломает scan matcher.

**Камера** (после замеров): `CAMERA_MOUNT_HEIGHT_M=0.125`, `CAMERA_MOUNT_FORWARD_M=0.12`
(объектив ≈12 см впереди центра вращения). `CAMERA_HFOV_DEG=60`.

## 6. Камера — depth-восприятие (важная часть, читать внимательно)

### Как устроено
Кадр 640×480 → Depth-Anything-V2 (ONNX, `models/depth_anything_v2_small.onnx`,
~95 МБ, НЕ в git) → метрическая depth-карта → обратная pinhole-проекция в мир
(`camera_perception/projection.py`, учитывает высоту/вынос/наклон камеры и позу
робота) → два независимых канала:
1. **В карту**: точки на высоте 3–50 см над полом → occupied conf 0.7. Видит
   НИЗКИЕ препятствия под плоскостью 2D-лидара. Одного попадания хватает, чтобы
   A* стал клетку объезжать.
2. **В реактивный слой** (`obstacle_distances_by_sector`): секторные дистанции;
   по фронту `min(лидар⊕УЗ, камера)` — камера может только сократить дистанцию,
   не объявить «свободно» в обход лидара.

### Что было сломано и почему камера «не отрабатывала»
В сохранённых картах камера записала ~1 ячейку (криминалистика по сигнатурам
log-odds: камера≈81.5, лидар≈90.5, УЗ≈68–77). Причины:
- **Серво камеры не инициализировались** на SD-системе → камера смотрела куда
  попало → точки проецировались на неверные высоты → фильтр 3–50 см выбрасывал всё.
  ИСПРАВЛЕНО: `_init_camera_servos()` ставит pan=90/tilt=25 при старте восприятия
  (через существующий `bot.Ctrl_Servo`, новых зависимостей нет).
- Высота была 0.10 вместо 0.125 (сдвиг на величину нижнего порога фильтра).
- Ленивая загрузка 95-МБ модели съедала старт коротких сессий.

«Робот бился, когда радиодатчик не ловил, а камера не отрабатывала» — наслоение:
протухание данных (инференс 1–3 с старил УЗ за таймаут), слепые зоны камеры
(<15 см, FOV ±30°), узкий конус УЗ, мусорный лидар. Большинство закрыто (§7–8).

### Доработки камеры этой сессии
- **Убран жёсткий потолок 2.5 м**: анализ сцены до `CAMERA_MAX_RANGE_M=6.0`
  (реактив + выбор направления видят всё изображение), запись в карту до
  `CAMERA_MAP_MAX_RANGE_M=3.0` (дальняя моно-глубина шумит, occupied-ячейка
  сразу блокирует A*).
- **Коридорная метрика** (`corridor_halfwidth_m=0.125`): фронт-дистанция учитывает
  точки в полосе шириной робота НЕЗАВИСИМО от пеленга — ловит угловые препятствия
  «под колесо» (пеленг >30°, в сектор не попадает, в полосу — да) и диагональные
  стены (ультразвук там слеп, эхо уходит вбок).
- **Страховочный канал `nearest_in_depth_band`**: ближайший объект по курсу прямо
  из depth-карты, без проекции и фильтра высот — робастный минимум в полосе у
  линии горизонта (пол там физически дальше ~3 м, ложных стопов нет). Ловит сетки/
  прозрачное/редкие облака, где секторный канал молчит. Вливается в `front` через min.
  ВАЖНО: если depth-модель «видит сквозь» конкретную сетку — это видно на отладочном
  кадре (полоса останется дальней); аппаратно такую сетку три сенсора не увидят.
- **Отладочные кадры** (`camera_perception/debug_viz.py`, флаг
  `CAMERA_DEBUG_SAVE_ENABLED=False`): раз в 3 с в `camera_debug/` JPEG — слева кадр,
  справа depth цветом с красной маской пикселей-препятствий (тот же боевой пайплайн)
  и секторными дистанциями + жёлтая рамка band-канала. Ротация 200 кадров.
- **Поза захвата кадра**: `update_with_camera_depth(robot_pose=...)` получает позу
  В МОМЕНТ ЗАХВАТА кадра (из `_DepthWorker`), а не потребления — иначе на ходу
  облако сдвигалось на путь за время инференса (нашёл ревью-агент).
- **Камере свой таймаут 1.5 с** в fusion + `invalidate_camera()` после поворота
  (секторы сняты в старом курсе, после доворота «front» смотрит вбок).

### Открытый вопрос камеры (проверить на железе)
Горизонтален ли взгляд при tilt=25°? Направить на стену, центр кадра должен быть
на высоте объектива (~12.5 см). Если смотрит вниз/вверх — вписать фактический угол
в `CAMERA_MOUNT_TILT_RAD` или подобрать другой `CAMERA_SERVO_TILT_DEG`.

## 7. Плавность движения и DWA

### Четыре доработки плавности
1. **Фоновый инференс** (`_DepthWorker`): нейронка (0.3–3 с) больше не блокирует
   контур управления; `_update_sensors` неблокирующе забирает последний результат.
   Не возвращать инференс в `_tick()`.
2. **Круиз**: `_safe_move_forward(stop_at_end=False)` не останавливает моторы между
   последовательными «вперёд»; стоп только при препятствии/повороте.
3. **Плавное замедление**: `_speed_for_distance` — линейный спад PWM от
   `SLOWDOWN_START_M=0.7` до `MIN_MOVE_PWM=22` у `min_obstacle_distance=0.4`.
4. **Подруливание** `forward_steer` (дифференциал бортов через `move_param_forward`):
   увод к простору на круизе (`_steer_to_free_space`, кап ±20%) и докрутка курса
   в legacy-навигации. Модель в одометрии: v×(1+|p|/200), ω=v·p/100/WHEEL_BASE.

### DWA (Dynamic Window Approach) — локальный планировщик
Был полностью реализован, но НЕ подключён (мёртвый код). Теперь ведёт робота в
режиме NAVIGATION: непрерывные дуги к waypoint'ам A*.
- `compute_velocity_command` **векторизован** (~1 мс на Pi вместо секунд):
  кандидаты динамического окна симулируются закрытой формой дуг, коллизии — по
  **непрерывному полю расстояний** `_clearance_field` (не бинарный inflated grid:
  у границ он дедлочил), скоринг progress/align/clearance/velocity.
- **escape-правило**: из тесного места (робот уже в inflated-зоне — шум позы на
  железе) можно выскальзывать вдоль границы, нельзя углубляться; смягчённый порог
  только на первых `min(0.4, 0.5·max_speed·predict_time)` м дуги.
- `car_adapter.move_tank(left,right)` — танковое управление; команда `tank` в
  одометрии с квантованием мёртвой зоны моторов.
- Реактивная страховка поверх: `DWA_BLOCK_FRONT_M=0.32` → стоп+объезд (критично в
  localization, где свежие препятствия в карту не пишутся), кап скорости по фронту
  `v_cap=0.5·(front−0.20)`, stall-детектор → перепланирование → (после
  `DWA_REPLAN_LIMIT=2`) объезд.
- **Откат одним флагом `DWA_ENABLED=False`** → старый дискретный автомат.

Константы DWA: `DWA_MAX_SPEED_M_S=0.16`, `DWA_MAX_YAW_RATE=1.2`, `DWA_STALL_LIMIT=12`,
`DWA_NO_DATA_TICKS=5`.

**Починен родной баг** `_simplify_path`: проверял отклонение от хорды к СОСЕДНЕЙ
точке, из-за чего схлопывал ЛЮБОЙ A*-путь в прямую «старт-финиш» — даже сквозь
стены. Заменён на line-of-sight (`_segment_free`).

## 8. Фиксы по многоагентному аудиту 4 флоу

**Построение карты:**
- Лидар пишется в карту только при НОВОМ обороте (`revolution_count`) и |ω|≤0.6 рад/с
  (один скан писался 2–3 раза → насыщение log-odds; при вращении точки оборота
  мажутся на 5–15°).
- Векторная запись `OccupancyGrid.update_cells_batch` (поштучный `update_cell` ел
  0.1–0.4 с/тик и просаживал контур; дубликаты ячеек внутри батча схлопываются).
- УЗ в карту ≤2 Гц; история позы `_HISTORY_MAXLEN` 1000→20000 (было ~50 с при 20 Гц);
  фикс алиаса позы в `load_map`; в localization likelihood field не пересчитывается зря.
- Известное ограничение (НЕ чинили, нужен отдельный слой): free-лучи лидара
  постепенно стирают камерные «низкие» препятствия из карты; защита — реактивный слой.

**Движения/одометрия:**
- `forward_steer` при p<0 занижал скорость вдвое (`move_param_forward` всегда
  усиливает борт) → формула v×(1+|p|/200).
- Порядок `_tick`: одометрия ПЕРЕД сенсорами (прайр scan matcher не отстаёт на тик).
- `turn_speed` 20→25 (было ниже порога трогания 22 → фантомные повороты в одометрии).
- `set_deflection`: `round` вместо `int` (паразитный страйф 30/29).

**Избегание препятствий:**
- Конфликт лидар/УЗ («лидар<0.2 & УЗ>0.4 → верим УЗ») только если близкие точки в
  конусе УЗ ±15° (иначе объект на пеленге 20° вне конуса — УЗ его не видит, «свободно»
  ничего не опровергает).
- Кластер ≥5 точек <12 см = прижатый объект (0.10 м), а не «простор» (раньше
  фильтр шума выкидывал прижатую стену → робот поворачивал В неё).
- Секторная дистанция = третья ближайшая точка (робастный минимум; медиана нижнего
  квартиля растворяла тонкие ножки стульев в точках дальней стены).
- Отъезды назад через `_safe_backward` (контроль сектора 'back' лидаром; раньше
  слепые).
- `critical_distance` 0.15→0.20 (мерится от центра; 0.15 = 3 см от носа камеры).

**Goto / движение к точке:**
- `DWA_BLOCK_FRONT_M` 0.28→0.32 (была «полоса замирания» квантования 0.28–0.30: кап
  обнулял команду раньше порога объезда → вечный застой/перепланирование).
- Терминал `_goal_unreachable`: в localization недостижимая цель (или 3 объезда на
  одном месте) → честный стоп + IDLE (раньше робот навсегда уходил в EXPLORATION,
  demo висел). В mapping остаётся исследование (карта пополняется).
- Лимиты `DWA_REPLAN_LIMIT` и повторных объездов; сброс DWA-состояния при всех
  перепланированиях.

**Проверки конфликтов моторов:** их НЕТ — `Ctrl_Car` и `Ctrl_Muto` в `Raspbot_Lib`
пишут в один регистр 0x01 одинаковым форматом. (Конфликт-кандидат существует только
на флешке — автостарт `raspbot.pyc` веб-демо дёргает моторы параллельно; на
SD-системе его нет.)

Вся регрессия (5 замкнутых сценариев A*+DWA, слияние сенсоров, модель увода) —
зелёная на синтетике.

## 9. Сегментация пола — обсуждение (НЕ внедрено, для будущего)

Пользователь рассматривает добавление определения пола сегментацией. Договорились:
оставить на потом, после полевой проверки текущего стека. Рекомендованный порядок:

**Шаг 0 (без новой нейронки): RANSAC-плоскость пола** по уже имеющемуся облаку
глубины. Пол — доминирующая плоскость z≈0, inliers=пол, что торчит=препятствие.
~5–10 мс numpy, ноль зависимостей, и БОНУС — автокалибровка наклона/высоты камеры
(закрывает открытый вопрос tilt=25°), фильтр высот станет адаптивным. Слабость:
доверяет глубине (сетку/стекло не объявит полом).

**Нейросетевые модели** (все на ADE20K, класс floor, без своего датасета):
- **SegFormer-B0** (`nvidia/segformer-b0-finetuned-ade-512-512`, 3.7M) — главный
  кандидат, экспорт в ONNX через HF Optimum, ~0.2–0.5 с @256² на Pi CPU, легко
  дообучить.
- TopFormer-Tiny (1.4M), SeaFormer-Tiny/Small, PP-MobileSeg, BiSeNetV2/Fast-SCNN —
  легче/быстрее, точность ниже.
- Промптовые (MobileSAM/FastSAM/CLIPSeg «floor») — медленно для реактива, но
  годятся для офлайн-разметки своего датасета.

**План внедрения**: RANSAC сейчас → замер SegFormer-B0 на железе на кадрах из
`camera_debug/` (256×256, латентность) → интеграция 3-м каналом (чередовать
кадры depth/seg в `_DepthWorker`, маска «не-пол на полу»→препятствия,
«пол»→фильтрация шума depth) → fine-tune 50–100 кадров если базовая путает.
Главный риск — CPU-бюджет (вторая нейронка уполовинит FPS depth), решать по
фактическим миллисекундам.

В репо есть deprecated `camera_segmentation/` (старая попытка на 46 кадрах, не
работала) и `update_with_camera_segmentation` в slam_core (вход для маски жив —
интеграция дешевле, чем кажется).

## 10. Что осталось — калибровка на железе (приоритет)

Полный чек-лист — `ROBOT_CHECKLIST.md`. Кратко, по приоритету:
1. **Замер диаметра колеса** → `WHEEL_DIAMETER` (пересчёт rpm→скорость, задний свес).
2. **Лидар**: `diagnose_lidar.py` (порт/скорость) → тест у стены в 1 м
   (`DISTANCE_DIVISOR`) → explore (зеркальность `CLOCKWISE`, поворот `ANGLE_OFFSET_DEG`).
3. **Одометрия**: `calibrate_odometry.py` → `MAX_LINEAR_SPEED`, `MAX_ANGULAR_SPEED`.
4. **Камера**: горизонтальность при tilt=25° → `CAMERA_MOUNT_TILT_RAD`.
5. **Контрольный explore**: стены не кольца, траектория не «замерзает», плавное
   движение без стоп-стартов, минимум «Конфликтов».
6. **goto**: DWA ведёт плавными дугами; предмет не на карте → реактивный стоп+объезд.

Критерий успеха всего: карта со стенами (не кольцами), живая SLAM-траектория,
плавная езда, доезд до цели.

### Стратегия защиты (из `/council`)
Не добавлять новые фичи — доказать работу на железе. Демо как спектакль с планами
А/Б/В (живой explore+goto → goto по готовой карте → видео+PNG). 30-сек трюк: низкая
коробка, которую не видит лидар, — камера останавливает. Графики для отчёта:
дрейф одометрии vs SLAM (`compare_trajectory.py`), loop-closure, ablation камеры.
Честный слайд «Ограничения»: нет энкодеров/IMU, нет глобального loop closure,
сегментация пола — следующий этап. Демо-логистика: репетиция в зале защиты, свежий
аккумулятор (просадка напряжения плывёт PWM-одометрию), люди вокруг «зарастят»
карту лидаром — иметь кнопку рестарта карты и видео-бэкап.
